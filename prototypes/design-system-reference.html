1.3 Spacing scale — strict 4px grid

Every padding, gap, and margin in the prototype must land on a 4px step: 4 · 8 · 12 · 16 · 20 · 24 · 28 · 32 · 36 · 40. Values like 6, 9, 10, 11, 14, 30 are not allowed — they were the accumulation that v1.0 inherited and v1.1 cleaned up. If a new component asks for "just a touch more" — round to the nearest grid step, never invent a half-step.

Hairline allowance. 1px / 2px / 3px are permitted ONLY for visual artifacts that aren't conceptually "spacing": borders (1px), accent bars (2px), divider lines, accent-dot glow halos, micro tag v-padding (e.g. .chip { padding: 1px 8px } where the 1 is intentional pill compactness). Hairlines are a different visual register and don't pollute the grid.
What's NOT on the 4px grid (by design). Type sizes follow a modular scale (10, 11, 12.5, 13, 14, 15) calibrated for legibility — they answer a different question and live on a separate scale. Same for radii (--radius-sm: 6, --radius-md: 10, --radius-lg: 14) — corner softness is a perceptual register, not a spacing register. Both are out of scope for the 4px-grid rule.
The single intentional exception. .kpi-card uses asymmetric padding: 16px 16px 6px — the 6px bottom is hand-tuned to sink the accent bar to the card's base as a "foundation line" (see §5.1). DASHBOARD-LOCAL only — no other component is allowed to break grid for aesthetic reasons.

Layout & geometry

Page-level grid, table edge caps, anchor-based column widths, and responsive padding. The edge-cap rule and the anchor system are the two most distinctive geometric contracts on the Dashboard — both must propagate to every other table page.

The Dashboard uses grid-2col (1fr + 320px aside) to split tables on the left and widgets on the right. --aside-w: 320px is the canonical right-aside width across the app.

Tables fix the leftmost / rightmost columns to a pixel anchor; middle columns share the remaining space proportionally by weight. Schema is fixed regardless of viewport — Order ID, datetime, Plan name always read fully, no matter how the panel resizes.

Sizing formula: anchor ≥ content_max + 16 (inner pad) + 32 (edge cap) = content_max + 48. For "22 Apr · 09:42" (113px) → 113 + 48 = 161 → rounded up to 164px (next 4-grid step) = --anchor-id. Same calculation for "ORD-10900" (110px) → 158 → 164. 164px gives both a comfortable fit while keeping centered content inside the row-divider boundary (which itself ends at the edge cap on each side). All anchor values land on the 4px grid: 144 (36×4), 164 (41×4), 168 (42×4).

    <col style="width: var(--anchor-l)">       /* 164px · Order ID */
    <col style="--w: 4">                       /* Client ID  · weight 4/19 */
    <col style="--w: 5">                       /* Plan       · weight 5/19 */
    <col style="--w: 3">                       /* Amount     · weight 3/19 */
    <col style="--w: 3">                       /* Payment    · weight 3/19 */
    <col style="--w: 4">                       /* Status     · weight 4/19 */
    <col style="width: var(--anchor-r)">       /* 164px · Created date */

A small JS helper, applyDtAnchors(), runs on resize / ResizeObserver / cold-load. It reads the table's --anchor-l, --anchor-r, and --col-total, then computes (wrapWidth − anchorL − anchorR) × w / total for each middle <col>. Anchored columns stay locked; middle columns redistribute precisely.

3.9 Bulk-action surface .bulk-bar added in v2.6

A horizontal action strip that appears between the filter bar and the data table whenever one or more checkbox rows are selected. Used on Orders / Renewals / Payments / Plans / Proxies. Two design-decisions define the pattern: matrix-driven button set (per-tab × per-row-state action visibility) and Set-based selection model (state lives in JS, not .chk.checked).

Markup contract.

<div class="bulk-bar" id="ordersBulkBar">
  <div class="bulk-summary">
    <span class="chk checked" onclick="clearOrdersSelection()"></span>
    <span id="ordersBulkCount">0 selected</span>
  </div>
  <div class="bulk-actions" id="ordersBulkActions"></div>
</div>

Action registry. Each page declares a flat XXX_ACTION_REGISTRY mapping label → metadata ({ mode: 'single'|'multi', handler, tone?: 'danger' }). One source of truth for what each bulk button does. The matrix XXX_BULK_ACTIONS then maps (tab, row-state) → action[] — e.g. Orders / Active tab exposes Suspend · Cancel · Edit, while Orders / New tab exposes Send reminder · Confirm payment · Cancel.

Row-state-aware filtering. On heterogeneous-tab pages ("All" / "Attention"), computeXxxLabels(ids) intersects the per-status label sets of every selected row — only the actions valid for ALL selected rows survive. Eliminates the "operator picks action that's only valid for some rows" failure mode.

Single-vs-multi gating. Buttons tagged data-bulk-mode="single" hide via display:none when n ≥ 2 rows are selected. The bar then shows just the count + the clear button. Used for actions that semantically operate on one row (e.g. Assign proxy, Inspect).

Selection clearing triggers. Always invoke clearXxxSelection() — never just clear DOM classes — on: tab change · filter change · successful bulk-action completion. Otherwise the Set drifts from the DOM.

3.10 Form validation added in v2.7

Three coordinated layers — HTML5 native attributes, shared CSS state, and a generic JS validator.

LayerWhat it doesUsed on
HTML5 attrs Every numeric field carries type="number" min="N" max="M" step="S" matching its semantic bounds (hours: 0–720, %: 0–100, USD cap: 0–99999, etc.) 27 numeric inputs across Plan Create · Plan Edit · Settings (Grace · Flags · API · Providers) · New Order modal
Required marker <span class="req">*</span> appended to .form-label text on fields that must have a value. Conservative scope — only fields where blank is semantically invalid. Identity fields · Plan name · Carrier · Region · Pool · Settings defaults · Quantity
Invalid state .form-input.invalid renders border-color: var(--danger). Toggled by the JS validator; cleared automatically when the user types a valid value. Reused from create-client validator; now generic across all forms.

Generic validator. validateNumericInputs(rootEl) walks every input[type="number"] in scope, reads its min/max attributes, and returns {el, label, value, reason}[]. Labels are extracted via cloned labelEl with all <span> children removed — a critical detail, because a naive textContent.replace(/i/g,'') would strip every letter "i" from the actual label text along with the help-tip icon. Empty values fail only if the field's label carries .req.

Domain rules layer. Section-specific Save handlers wrap the generic validator and add per-section checks. Grace Rules: monotonicity (Second reminder < Pre-renewal, Third < Second). System Flags: bounds-only. Provisioning rule: duplicate-guard on (carrier, region).

Error surfacing. _toastValidationErrors(errors) emits a warn-tone toast: title "1 field needs attention" / "3 fields need attention", body lists up to 3 specific errors verbatim ("Default grace period (hours): Required", "Discount cap: Max 100"), then "+ N more" collapse. Plural noun + verb agree with the count.

§4

Patterns

Cross-cutting conventions about how components are used, not just what they look like. These are the rules that make the prototype feel coherent — they're not enforceable by CSS alone, they live in markup and in the data layer.

4.1 Entity-link convention · Client Identity Rule

Every clickable ID that opens a detail page renders blue via .td-link, .client-link, or .plan-link. Names render as plain secondary text (.muted or .t-primary) even when the row navigates somewhere — visual consistency across identity columns wins.

Entity links inside a .col-id cell already inherit mono via the .dt cascade. Outside the table — in a kv-row, timeline, breadcrumb, modal — wrap the entity link in .kv-val.mono (kv-row) or apply the .mono utility directly (e.g. <a class="client-link mono">${'CLI-2184'}</a>) so the ID renders mono + tabular-nums everywhere. Plan / Carrier / Pool names that happen to live inside a .td-link stay sans-serif — they're text labels, not identifiers.

The Clients list is the single approved exception: the human name remains the primary browsing affordance because that surface is people-browsing, not entity-browsing. Client ID still sits in its own column for copy-on-click.

4.2 Header naming convention

Append "ID" to the column header only when the column shows the literal entity ID (an entity link). Chip-status columns keep the bare noun.

4.4 Identifier & numeric typography

Mono + tabular-nums applies to every identifier-shaped value — not just numeric columns. 4.7 Policy-override hatch — centralized default + per-entity opt-out added in v2.7

When an operational policy applies to multiple entities by default, but a small minority need a custom value, separate where the policy lives (centralized) from where the exception lives (per-entity, opt-in). The default is inherited; the override is one .toggle-v2 + one conditional field on the entity-edit page.

Canonical instance — Settings → Provisioning ⇄ Plan Edit. Settings → Provisioning hosts a .dt table mapping (Carrier · Region) → default Pool (+ optional fallback chain + auto-assign toggle). Plan Edit → Infrastructure exposes an "Override default pool" toggle that:

  • When OFF (default) — shows a disabled readout populated from getProvisioningDefault(carrier, region).defaultPool. Pool select is hidden.
  • When ON — hides the readout, reveals the editable pool <select>. Stores plan.poolOverride = true + plan.pool = '<chosen>'.

Why this shape. The centralized table is the single source of truth — most plans inherit, so changing a default propagates without per-plan edits. The hatch is opt-in: the operator must explicitly take responsibility for diverging from policy. The Plan Edit surface stays clean for the 95% inherit-default case.

When to reach for this pattern. Any operational knob where (a) a sensible group-default exists, (b) an entity can opt out, and (c) the override is exceptional rather than typical. Other future candidates: refund-policy override per VIP client, rate-limit override per API key, grace-period override per plan tier.

Anti-pattern. Don't duplicate the policy editor onto every entity-edit page. If every plan editor shows a full pool dropdown by default, operators end up making per-plan decisions that should be group-policy decisions, and changing the group default requires touching every plan.

6.5 Standing invariants

  • Bulk-bar geometry. .col-chk = 32px width + 20px pad-left composes with --table-edge-cap. All bulk-bar pages share this column.
  • Detail-page .kv-row rhythm uses --row-divider-inset: 20px, intentionally different from the table's 32px edge cap. Do not change.
  • Renewals per-bucket Status. Status column content is context-dependent on the active bucket (24h/3d/7d/grace/expired/renewed). Renderer stays as-is.
  • Plans Capacity State precompute. Renderer always emits a derived state (no null fallback); the chip column is unconditional.
  • Bidirectional order ↔ proxy links. Every active assignment carries order.proxyId + proxy.currentOrderId + an ASN-<ord-suffix><letter> assignment row. validatePrototypeData() flags any drift.
  • Settings persisted state. All toggles read from / write to db.settings; DOM is a mirror, not the source of truth. syncXxxToggles() runs on every tab open.
